# EDID Design Notes

This repository does not include production EDID generation or application
source. It also omits device discovery, capture-card identifiers, register
addresses, sysfs paths, and board-specific commands.

A safe high-level design is:

1. Define a display profile as resolution, refresh-rate families, color
   capabilities, audio capabilities, and transport limits.
2. Build an EDID base block from a standards-compliant template.
3. Add a CTA-861 extension only for capabilities required by the profile.
4. Randomize identity-only fields when appropriate. Do not randomize timing or
   transport fields independently; they must remain internally consistent.
5. Recompute each 128-byte block checksum.
6. Parse the generated bytes with a second, independent parser and reject the
   result unless all advertised modes and checksums match the requested
   profile.
7. Pass the validated bytes to a platform-specific `EdidApplier` abstraction.
8. Re-read the active state, verify the source enumerates the intended modes,
   and roll back to the previous known-good EDID on failure.

Illustrative pseudocode:

```text
profile = choose_profile()
base = build_base_block(profile.identity, profile.base_modes)
cta = build_cta_extension(profile.video, profile.audio, profile.transport)
edid = finalize_checksums(base + cta)

if not independent_parser.validate(edid, profile):
    fail_closed()

previous = applier.snapshot()
if not applier.apply(edid) or not applier.verify(profile):
    applier.restore(previous)
```

The platform-specific implementation should be kept outside the portable
pipeline because applying malformed EDID data can disrupt display negotiation
or leave a source with no usable mode.
